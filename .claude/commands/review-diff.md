# Review Diff

## 触发条件

当用户要求 review 代码变更、检查 diff、查看是否有问题时执行。

## 步骤

### Step 1: 获取变更概览

```bash
git diff HEAD --stat
git diff HEAD
```

### Step 2: 逐文件分析

对每个变更文件，按以下维度检查：

#### 2.1 正确性

| 检查项 | 说明 |
|--------|------|
| 逻辑完整性 | 所有分支都被正确处理？有没有遗漏的 case？ |
| 边界情况 | 空列表、null、并发、索引越界 |
| 数据一致性 | DB 操作后 UI 是否会正确刷新？缓存是否需要清除？ |
| 类型安全 | 强制转换（`as`）是否安全？ |

#### 2.2 线程安全

| 检查项 | 说明 |
|--------|------|
| 共享可变状态 | 是否需要 `@Volatile` / `ConcurrentHashMap` / `Mutex`？ |
| 协程作用域 | IO 操作是否在 `launchIO` / `withIOContext` 中？ |
| 不可取消操作 | 删除等操作是否使用 `launchNonCancellable`？ |

#### 2.3 架构合规

| 检查项 | 说明 |
|--------|------|
| 依赖方向 | 是否违反模块间依赖规则？ |
| 分层 | UI 层是否包含业务逻辑？ScreenModel 是否引用 UI 类？ |
| KMP | expect/actual 是否同步？ |

#### 2.4 UI/UX

| 检查项 | 说明 |
|--------|------|
| 状态覆盖 | loading / empty / error / content 四种状态都处理了？ |
| 选中模式 | 多选模式下的交互是否正确？ |
| 国际化 | 新增文案是否在 strings.xml 中？ |

#### 2.5 DI 与注册

| 检查项 | 说明 |
|--------|------|
| 新增依赖 | 构造函数新增的 `= Injekt.get()` 是否在 DI 模块中注册？ |
| Repository | 新增 Repository 方法是否接口和实现都有？ |

### Step 3: 输出报告

按严重程度分类：

- **Bug**：会导致崩溃、数据丢失或功能异常
- **问题**：逻辑不严谨但不会立即出问题
- **建议**：可以改进但不影响功能

对每个问题给出：文件:行号、问题描述、修复建议。

### Step 4: 编译验证

```bash
./gradlew :app:compileDebugKotlin
```
